What is S.E.A?

The S.E.A System is a multi-role emergency response platform built across a Flutter mobile app and a MERN web dashboard. It supports six roles — Users, Helpers, Ambulances, Police, Firefighters, and Admins — each with a distinct, purpose-built experience.

At its core is a smart dispatch engine that automatically identifies the nearest responder within a 5 km radius, enforces a 10-minute response window, and escalates intelligently if no one responds.

Goals

The S.E.A system is designed around five primary goals that drive every architectural decision.

Primary Goals
01
Reduce emergency response time. Get the right help to the right person as fast as possible by automating dispatch and eliminating manual coordination.
02
Make emergency assistance accessible. Any registered civilian can summon help in seconds, regardless of technical literacy, with as few as 3 taps or a single button press.
03
Maximize responder utilization. Ensure no available responder within range is missed, and no responder is overloaded with multiple simultaneous assignments.
04
Build a smart escalation safety net. No emergency goes unanswered. If the primary responder can't make it, the system automatically finds the next best option.
05
Establish accountability at every step. Every action from alert to resolution is timestamped and logged, creating a transparent, auditable record of every incident.
Secondary Goals
06
Empower civilians as first responders. By bringing helpers (cabs, bikes) into the network, ordinary people with vehicles become part of the emergency response ecosystem.
07
Give admins full situational awareness. A real-time dashboard means no incident goes unnoticed and system health is always visible.
08
Build a foundation for scale. The architecture should be ready to onboard trusted organizations like Rescue 1122 and Edhi Foundation without a system redesign.
09
Earn public trust. Role-based access, strict dispatch rules, and zero data leakage between roles ensures the platform is safe, fair, and trustworthy.

Architecture

Three independently runnable applications inside a single pnpm monorepo, sharing no direct database access.

System Boundaries

The Flutter mobile app serves all roles except admin. The React web dashboard serves admins only. The Express server is the single backend for both clients — it owns all business logic, dispatch decisions, and data. No client ever talks directly to the database.

Auth & Access Model

Authentication is handled entirely by the Express server using JWT with role claims. On login, the server issues a signed JWT containing the user's ID and role. Every subsequent REST request must include this token in the Authorization: Bearer <token> header.

🔒

JWT role is the single source of truth for access. No client-side role check is trusted. The server re-verifies role on every request and every socket event, regardless of what the client claims. There is no refresh token in v1 — session expiry forces re-login.

Architecture Invariants
1
No business logic in controllers or route handlers. Controllers validate input, call a service, and return a response. All dispatch decisions, escalation timers, radius calculations, and role-restriction rules live exclusively in the service layer.
2
No client ever writes to or reads from the database directly. All data access goes through the Express server.
3
No request handler runs long-lived background work. Escalation timers and scheduled tasks are managed by a dedicated background service. A controller must hand off async work before returning a response.
4
Socket event names are never hardcoded as strings. All event names are defined in packages/shared/src/events.ts and imported wherever used.
5
Every incident action is logged before a response is sent. No state change is communicated to any client unless it has first been persisted to the database.
6
Helpers are never dispatched to crime or fire incidents. This rule is enforced in the dispatch service, not the route or controller. No API call or socket event can bypass it.
7
JWT role is the single source of truth for access. The server re-verifies role on every request and every socket event, regardless of what the client claims.

Helper Flow

🚫

Strict restriction: Helpers are never dispatched to crime or fire incidents. This is enforced at the dispatch service level — no API call can bypass it.

Code Standards

Non-negotiable rules enforced across every language and framework in the stack.

TypeScript — Universal Rules
🚫

No any — ever. Use unknown and narrow it, or define a proper type. No type assertions (as SomeType) unless absolutely unavoidable.

TypeScriptZod-inferred types — preferred pattern
// ✓ Correct — types derived from Zod schema const incidentSchema = z.object({ id: z.string(), type: z.enum(['medical', 'fire', 'crime']), status: z.enum(['active', 'resolved']), location: z.object({ lat: z.number(), lng: z.number() }), createdAt: z.date(), }) type Incident = z.infer<typeof incidentSchema> // ✗ Wrong — never define type separately from schema type Incident = { id: string; type: string; ... }

MongoDB — Schema Rules
→
One schema per file — filename matches model name (incident.model.ts)
→

→
Transactions used for any operation writing to more than one collection
